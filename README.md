## Как дургим участникам смотреть какие шаги выполнены?
-каждый шаг из **implementation-plan.md** оформлен как отдельный Issue.
- сортируете по времени(сначала старые)
- Open, In progress, Closed смотрите
- Ставить теги для того, чтобы другие участники понимали
- Подождать выполнение шага, если следующий шаг зависит от текущего(или взять другую независимую задачу)



# Project Workflow

## 1. Какие документы есть в проекте

- **ТЗ по ГОСТ** → формальные требования
- **docs/specs/Lecture-Notes-System.md** → архитектура и общая модель системы
- **docs/specs/implementation-plan.md** → задачи для GitHub Issues, Codex и работы команды

---

## 2. Что использовать для чего

- **ТЗ** — отвечает на вопрос: **что нужно сделать**
- **Lecture-Notes-System.md** — отвечает на вопрос: **как устроена система**
- **implementation-plan.md** — отвечает на вопрос: **что делать по шагам**

---

## 3. Как брать задачу в работу

Перед началом любой задачи нужно:

1. Посмотреть, есть ли требование в ТЗ
2. Найти связанный раздел в `Lecture-Notes-System.md`
3. Найти нужный шаг в `implementation-plan.md`
4. Создать отдельную ветку
5. Только после этого приступать к коду

---

## 4. Как работать с AI

Не нужно отправлять в AI всё ТЗ целиком.

В промпт нужно передавать:

- краткий контекст задачи
- конкретный шаг из `implementation-plan.md`
- какие файлы или модули нужно изменить
- критерии готовности

### Правильно
- делать одну маленькую задачу за раз

### Неправильно
- просить сразу реализовать всю систему
- отправлять огромный документ без пояснений

---

## 5. Как работать с ветками

Формат:

- `feature/...` — новый функционал
- `fix/...` — исправление ошибок
- `docs/...` — документация

Примеры:

- `feature/auth`
- `feature/upload`
- `feature/transcription`
- `fix/login-validation`
- `docs/workflow`

---

## 6. Как делать коммиты

Коммиты должны быть короткими и понятными.

---

## 7. Как делать Pull Request

В каждом Pull Request нужно указать:

- что сделано
- какой шаг `implementation-plan.md` реализован
- что нужно проверить

### Пример

**Что сделано**  
Добавлена загрузка лекции и сохранение записи в БД.

**Шаг**  
`implementation-plan.md → Upload lecture`

**Проверка**
- можно загрузить файл
- создаётся запись в БД
- статус сохраняется корректно

---

## 8. Основной принцип работы

1. Сначала смотрим требование
2. Потом смотрим архитектуру
3. Потом берём конкретный шаг
4. Потом пишем код
5. Потом открываем PR
6. Потом проходим review
